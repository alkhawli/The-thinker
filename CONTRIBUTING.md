# Contributing to The Thinker

Thank you for your interest in contributing to The Thinker! This document provides guidelines for contributing to the project.

## Ways to Contribute

### 1. Enhance Input Documents
- Add new documentation types
- Improve existing documents
- Add real-world examples
- Fix inaccuracies or outdated information

### 2. Share Generated Ideas
- Submit valuable ideas generated by the agent
- Document implementation outcomes
- Share success stories

### 3. Improve Documentation
- Fix typos and grammar
- Add clarifications
- Create tutorials or guides
- Translate documentation

### 4. Refine the Agent
- Improve agent instructions
- Add domain-specific knowledge
- Enhance agent capabilities
- Optimize agent behavior

### 5. Create Templates
- Add new idea templates
- Create document templates
- Design workflow templates

## Contribution Process

### For Documentation Changes

1. **Fork the repository**
   ```bash
   git clone https://github.com/alkhawli/The-thinker.git
   cd The-thinker
   ```

2. **Create a branch**
   ```bash
   git checkout -b improve-documentation
   ```

3. **Make your changes**
   - Edit the relevant files
   - Follow existing style and format
   - Test your changes

4. **Commit your changes**
   ```bash
   git add .
   git commit -m "docs: improve quickstart guide clarity"
   ```

5. **Push and create PR**
   ```bash
   git push origin improve-documentation
   ```
   Then create a Pull Request on GitHub

### For Input Documents

1. **Check existing documents**
   - Review `input/README.md` for guidelines
   - Ensure your document doesn't duplicate existing content

2. **Create your document**
   - Use clear markdown formatting
   - Include a descriptive title
   - Add a brief overview
   - Structure with headings

3. **Test with the agent**
   ```
   @the-thinker Analyze the new [document-name].md
   ```

4. **Submit for review**
   - Create a PR with your new document
   - Explain what it adds to the knowledge base
   - Show example agent interactions

### For Agent Configuration

1. **Discuss first**
   - Open an issue describing the proposed change
   - Explain the rationale
   - Get feedback from maintainers

2. **Make changes carefully**
   - Agent behavior changes affect all users
   - Test thoroughly before submitting
   - Document the changes

3. **Provide examples**
   - Show before/after behavior
   - Include test queries
   - Demonstrate improvements

## Commit Message Guidelines

Follow the [Conventional Commits](https://www.conventionalcommits.org/) specification:

- `feat:` New feature
- `fix:` Bug fix
- `docs:` Documentation changes
- `style:` Formatting, missing semicolons, etc.
- `refactor:` Code restructuring
- `test:` Adding tests
- `chore:` Maintenance tasks

Examples:
```
docs: add example for API integration
feat: add new security policies document
fix: correct typo in quickstart guide
refactor: reorganize input documents
```

## Style Guidelines

### Markdown Style

- Use ATX-style headers (`#` not underlines)
- Include blank lines between sections
- Use code blocks with language tags
- Keep lines under 100 characters when possible
- Use bullet points for lists

### Documentation Style

- Write in clear, concise English
- Use active voice
- Include examples
- Define technical terms
- Keep paragraphs short (3-5 sentences)

### Code Examples

- Use proper syntax highlighting
- Include comments for complex logic
- Show both correct and incorrect usage
- Keep examples realistic and practical

## Input Document Guidelines

### Structure
```markdown
# Document Title

## Overview
Brief description

## Main Sections
Content organized logically

## Examples
Concrete examples

## References
Links to related documents
```

### Best Practices

1. **Clarity**
   - Use simple language
   - Define acronyms
   - Explain context

2. **Completeness**
   - Cover all relevant aspects
   - Include edge cases
   - Provide examples

3. **Consistency**
   - Match existing document style
   - Use consistent terminology
   - Follow naming conventions

4. **Accuracy**
   - Verify all information
   - Keep content current
   - Cite sources when needed

## Pull Request Process

1. **PR Title**
   - Use conventional commit format
   - Be descriptive but concise
   - Example: "docs: add API security guidelines"

2. **PR Description**
   - Explain what changes and why
   - Reference related issues
   - Show examples of improvements
   - Include screenshots if relevant

3. **Review Process**
   - Respond to feedback promptly
   - Make requested changes
   - Ask questions if unclear
   - Be respectful and constructive

4. **Merging**
   - PRs require approval from maintainers
   - All checks must pass
   - Conflicts must be resolved
   - Squash commits when merging

## Testing Your Changes

### For Documentation
- Proofread carefully
- Check all links work
- Verify code examples
- Test with the agent if applicable

### For Input Documents
```
@the-thinker Analyze [your-document].md
@the-thinker What insights can you draw from [your-document].md?
@the-thinker Generate ideas based on [your-document].md
```

### For Agent Configuration
- Test with various prompts
- Compare behavior before/after
- Verify no regressions
- Document behavioral changes

## Questions or Issues?

- **Questions:** Open a discussion on GitHub
- **Bugs:** Open an issue with details
- **Feature Requests:** Open an issue with use cases
- **Security:** Email maintainers directly (see README)

## Code of Conduct

### Be Respectful
- Treat everyone with respect
- Welcome newcomers
- Accept constructive criticism
- Focus on what's best for the project

### Be Constructive
- Provide helpful feedback
- Explain reasoning
- Suggest improvements
- Share knowledge

### Be Professional
- Keep discussions on-topic
- Avoid inflammatory language
- Respect different perspectives
- Maintain confidentiality when needed
